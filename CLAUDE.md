# NRS Repo — Agent Guide

This repo *is* the NRS framework. It is not a project that uses NRS, it is the one that defines it. The root has no `project.context.md` / `team.context.md` / `corporate.context.md`, so this `CLAUDE.md` is hand-written (not generated by `nrs generate claude`). Do not add the `DO NOT EDIT — generated by NRS` header to this file.

## What NRS Is

A layered context framework for agentic codebases. Canonical spec in [SPEC.md](SPEC.md); roadmap in [ROADMAP.md](ROADMAP.md); academic backing in [references/](references/).

When the user asks about NRS semantics (layers, reference rules, size limits, validators, gap reporting, observe signals), consult `SPEC.md` — it is the source of truth and takes precedence over memory or prior conversations.

## Repo Layout

- `cli/` — Rust CLI (`nrs` binary). Subcommands: `generate`, `validate`, `init`, `install`, `claude`, `gap`. Entry point at `cli/src/main.rs`.
- `skill/nrs-setup/`, `skill/nrs-fix/` — Claude Code skills shipped with NRS.
- `docs/` — Astro documentation site (consumed at the public NRS website).
- `examples/hello-world/` — full sample Next.js project with context files at every layer. Its `CLAUDE.md` is **generated** — regenerate with the CLI, never hand-edit.
- `examples/formats/` — standalone examples of each `*.context.md` type.
- `references/` — cited PDFs (numbered `[N]-…`). `SPEC.md` cites these by `[N]`.

## Commands

Run from repo root unless noted.

- `cargo build --manifest-path cli/Cargo.toml` — build the CLI
- `cargo test --manifest-path cli/Cargo.toml` — run CLI unit + integration tests (in `cli/tests/`)
- `cargo run --manifest-path cli/Cargo.toml -- <subcommand>` — invoke the CLI without installing
- `cargo install --path cli` — install the `nrs` binary locally
- `nrs generate claude --dir examples/hello-world` — regenerate `examples/hello-world/CLAUDE.md` after editing its layer context files
- `nrs validate --dir examples/hello-world --strict` — validate example context files
- From `docs/`: `bun install` then `bun run dev` / `bun run build` for the docs site

## Working in This Repo

- **Spec is canonical.** If behavior in the CLI disagrees with `SPEC.md`, align them deliberately — don't silently drift. When adding a CLI feature backed by research, cite the paper number (`[N]`) the way `SPEC.md` does.
- **Generated files are generated.** Never hand-edit any `CLAUDE.md` carrying the `DO NOT EDIT — generated by NRS` header. Change the source context files, then run `nrs generate claude`. The root CLAUDE.md of *this* repo is the one exception and has no header.
- **Validators and generators are symmetric.** Changes to context file rules land in both `cli/src/validate/` and the relevant generator under `cli/src/generators/`, plus coverage in `cli/tests/`.
- **Test discipline matches what NRS preaches.** Evidence-based bug fixes: write the failing test first. No flaky tests — if a detector fires intermittently, increase input volume until deterministic.
- **Context file writing standards** (apply when editing anything under `examples/` or writing prose in `SPEC.md`): concise, dense, no source file paths in `*.context.md`, no duplication, no speculative examples. Apply the refactoring test — if a behavior-preserving refactor would break the line, it belongs in code, not context.

## Proposing Changes

Propose first, act after. For non-trivial edits (new CLI subcommand, new validator, spec change, new generator target), lay out the plan and wait for alignment before writing code. Spec edits especially — `SPEC.md` is load-bearing for every downstream generator, skill, and example.

## Commits

Gitmoji convention. Keep messages neutral and descriptive of the change itself; don't narrate the authoring process.
